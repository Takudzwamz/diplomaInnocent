## Как работает рекомендательная система

### Поток данных (упрощённо):

1. **Пользователь взаимодействует с товаром** (просмотр, клик, корзина, покупка)  
   → Записывается в таблицу `UserInteractions`

2. **Система определяет стратегию рекомендаций для пользователя:**
   - Если пользователь в A/B эксперименте → берётся стратегия из `ABTestAssignments`
   - Если нет → используется стратегия по умолчанию (Adaptive)

3. **Генерируется список рекомендаций** по выбранной стратегии:
   - **Popular** — топ товаров по продажам/просмотрам
   - **Collaborative Filtering** — «пользователи, похожие на вас, купили…»
   - **Content-Based** — косинусное сходство эмбеддингов товаров
   - **Adaptive (гибридная)** — `0.40·CF + 0.35·CB + 0.15·Trending + 0.10·Recency`

4. **Фиксируются события рекомендаций** (показ, клик, покупка) для offline-оценки  
   → Таблица `RecommendationEvents`

### Ключевые файлы:

| Что | Где |
|-----|-----|
| Адаптивный алгоритм | `Infrastructure/Services/AdaptiveRecommendationService.cs` |
| A/B тестирование | `Infrastructure/Services/ABTestService.cs` |
| Метрики | `Infrastructure/Services/RecommendationMetricsService.cs` |
| Offline-оценка | `Infrastructure/Services/OfflineMetricsService.cs` |
| Эмбеддинги (Azure OpenAI) | `Infrastructure/Services/ProductEmbeddingService.cs` |
| AI-рекомендации (cosine) | `Infrastructure/Services/AIRecommendationService.cs` |
| Сидирование данных | `Infrastructure/Data/StoreContextSeed.cs` |

## Почему именно такая архитектура

| Решение | Обоснование |
|---------|-------------|
| **Clean Architecture (3 слоя)** | Core не зависит от Infrastructure → можно менять БД или AI-провайдера без изменения бизнес-логики |
| **Razor Pages (не SPA)** | Монолитный проект проще развернуть и поддерживать; SSR лучше для SEO; для диссертации не нужен React/Angular |
| **.NET 10** | Кроссплатформенность, высокая производительность, встроённый DI, Entity Framework для ORM |
| **SQL Server** | Полнотекстовый поиск, надёжные транзакции, бесплатная Express-версия для разработки, Docker-образ для Linux |
| **Redis** | Кэширование рекомендаций (избежание повторного вычисления для одного пользователя за сессию) |
| **Детерминистический хеш (A/B)** | `MD5(userId + experimentName)` — один и тот же пользователь всегда в одной группе, не нужна предварительная таблица назначений |
| **COD (оплата при получении)** | Не требует сторонних API-ключей для работы; студент может проверить полный цикл заказа без платёжного шлюза |
| **Seed с фиксированным Random(42)** | Воспроизводимость: каждый запуск даёт идентичные 75 заказов, 895 событий, одинаковые CTR |

## Как работает A/B тестирование

1. Создаётся эксперимент (контроль: Popular, эксперимент: Adaptive, трафик: 50%)
2. При первом запросе рекомендаций пользователь назначается:
   ```
   bucket = MD5(userId + experimentName) mod 100
   если bucket < trafficPercentage → Treatment (Adaptive)
   иначе → Control (Popular)
   ```
3. Все последующие рекомендации используют назначенную стратегию
4. Каждый показ/клик/покупка записывается как `RecommendationEvent`
5. Админ-панель считает CTR, конверсию и статистическую значимость в реальном времени

## Как считаются метрики

### Online-метрики (из событий):
- **CTR** = клики / показы × 100%
- **Click→Cart** = добавления в корзину / клики × 100%
- **Cart→Purchase** = покупки / добавления в корзину × 100%

### Offline-метрики (ретроспективная оценка):
- **Precision@K** — доля релевантных среди top-K рекомендаций
- **Recall@K** — доля найденных релевантных из всех возможных
- **NDCG@K** — учитывает позицию (релевантный товар выше = лучше)
- **MRR** — средний ранг первого релевантного товара
- **Coverage** — доля каталога, охваченная рекомендациями

### Статистические тесты:
- **z-тест** для сравнения пропорций (CTR между группами)
- **t-тест Уэлча** для сравнения средних (AOV между группами)
- **p < 0.05** → разница статистически значима

## Как генерировать/изменять диаграммы

Все диаграммы написаны на языке [Mermaid](https://mermaid.js.org/) и находятся прямо в `.md`-файлах как блоки кода.

### Способ 1: Онлайн-редактор
1. Откройте [mermaid.live](https://mermaid.live) или [mermaid.ai](https://mermaid.ai/)
2. Скопируйте содержимое блока ` ```mermaid ... ``` ` из любого документа
3. Отредактируйте визуально → скачайте PNG/SVG

### Способ 2: Автоматическая генерация
```bash
cd diplomnaya-rabota
python generate_diagrams.py        # генерирует PNG в папку graphs/
python generate_diagrams.py --svg  # SVG-формат
```
Скрипт обрабатывает все `.md`-файлы и использует бесплатный API kroki.io.

### Способ 3: VS Code
Установите расширение **"Markdown Preview Mermaid Support"** — диаграммы отрисуются прямо в предпросмотре Markdown.

### Смена темы оформления:
Добавьте в начало блока:
```
%%{init: {'theme': 'forest'}}%%
```
Темы: `default`, `forest`, `dark`, `neutral`.

## Частые вопросы

### «Почему данные одинаковые при каждом запуске?»
Потому что `new Random(42)` — фиксированный сид. Это гарантирует воспроизводимость эксперимента.

### «Как запустить без Docker?»
Нужен SQL Server (Express или LocalDB) и Redis. См. [07-RUNNING-LOCALLY.md](07-RUNNING-LOCALLY.md).

### «Как добавить новую стратегию рекомендаций?»
1. Создайте класс, реализующий `IRecommendationStrategy` в `Infrastructure/Services/`
2. Зарегистрируйте в DI (`Program.cs`)
3. Добавьте значение в enum `RecommendationStrategy` в `Core/Entities/`

### «Как изменить веса гибридной модели?»
Файл `Infrastructure/Services/AdaptiveRecommendationService.cs`, константы:
```csharp
CF_WEIGHT = 0.40
CB_WEIGHT = 0.35
TRENDING_WEIGHT = 0.15
RECENCY_WEIGHT = 0.10
```

### «Нужен ли Azure OpenAI для работы?»
AI включён по умолчанию (`AI_Enabled = true`). Рекомендации работают на statistical/collaborative методах. AI добавляет Content-Based через эмбеддинги (Azure OpenAI). Если ключ недоступен — система автоматически переключается на фолбэк (рекомендации по бренду/типу).

### «Как посмотреть результаты эксперимента?»
Админ-панель → Recommendations → вкладки Overview и Evaluation. Данные обновляются автоматически из сидированных событий.
